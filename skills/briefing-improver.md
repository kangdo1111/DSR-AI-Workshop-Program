# 🔄 Briefing Improver — 통합 스킬

**목표**: 산출물을 한 번에 검증·개선·최적화

---

## 작동 순서

### ① Issue-Writer Phase
**목표**: 산출물의 약점을 발견하여 GitHub 이슈로 등록

**입력**: 리포트, 코드, 문서 등 산출물
**출력**: GitHub 이슈 목록 (ISSUES.md)

```
산출물 읽음
    ↓
검증 체크리스트 실행
    ↓
문제점 도출
    ↓
ISSUES.md 갱신
```

**확인사항**:
- [ ] 기능 정상 작동 여부
- [ ] 누락된 기능
- [ ] 문서화 완전성
- [ ] 코드 품질
- [ ] 신뢰성 (출처, 검증)

---

### ② Issue-Runner Phase
**목표**: 등록된 이슈를 분석하여 코드 수정 및 테스트

**입력**: GitHub 이슈 (Issue #N)
**출력**: 커밋 + 테스트 통과

```
이슈 분석
    ↓
현황 댓글 작성
    ↓
코드 수정 (branches/issue-N)
    ↓
테스트 실행
    ↓
Git 커밋
```

**각 이슈당**:
1. 문제 파악
2. 수정 전략 수립
3. 코드 변경
4. 테스트 검증
5. 커밋 메시지 작성

---

### ③ Doc-Optimizer Phase
**목표**: 모든 문서를 최신 상태로 갱신

**입력**: 수정된 코드 + 이슈 변경사항
**출력**: 최적화된 문서

```
CLAUDE.md 갱신 (개발 노트)
    ↓
SOUL.md 갱신 (프로젝트 목표)
    ↓
README.md 갱신 (사용자 가이드)
    ↓
ISSUES.md 상태 업데이트
```

**갱신 항목**:
- 버전 정보 (v1.0 → v1.1 등)
- 완료된 이슈 마크 (⏳ → ✅)
- 최근 커밋 해시
- 마지막 업데이트 날짜
- Phase 진행 상황

---

## 사용 예시

```
/briefing-improver

또는

/briefing [리포트 파일명]
```

### 한 번 실행 시
1. 산출물 검증 (issue-writer)
2. 발견된 이슈 모두 처리 (issue-runner)
3. 문서 전체 동기화 (doc-optimizer)

---

## 실행 조건

### 필수
- ✅ Git 저장소 초기화 완료
- ✅ .env 파일 존재 (API 키)
- ✅ GitHub 연동 완료

### 권장
- 커밋 전에 실행 (깨끗한 상태)
- 주간 1회 정기 실행
- 새 기능 추가 후 실행

---

## 예상 소요 시간

| Phase | 소요 시간 | 비고 |
|-------|---------|------|
| Issue-Writer | 5~10분 | 검증만 |
| Issue-Runner | 30~60분 | 이슈 개수에 따라 |
| Doc-Optimizer | 5~10분 | 문서 갱신 |
| **합계** | **40~80분** | 이슈 4개 기준 |

---

## 최종 아웃풋

✅ 모든 단계 완료 후:
```
- ISSUES.md: 최신 상태로 갱신
- CLAUDE.md: 개발 노트 업데이트
- SOUL.md: 프로젝트 목표 반영
- README.md: 사용자 가이드 최신화
- Git: 모든 변경사항 커밋 완료
- GitHub: 이슈 상태 동기화
```
